Embedding Pipeline Report
=========================

Scope
-----
This report documents how a frame travels from RTSP stream input to ReID embedding generation in this codebase. It focuses on capture, per-frame detection/tracking handoff, crop selection, preprocessing, backend inference, and embedding postprocessing.

Relevant Entry Points
---------------------
1) local test runner: localtest/run_localtest.py
   - Loads YAML config via crosscamreid.config.load_config(...)
   - Calls crosscamreid.pipeline.run_app(config)

2) production runner: production/app.py
   - Same run_app(...) path as localtest

Therefore localtest and production share the same embedding pipeline implementation in production/src/crosscamreid/*.


A) RTSP Capture: How Frames Are Fetched
---------------------------------------
Implementation: production/src/crosscamreid/capture.py (class RTSPCapture)

1. Object setup
   - Inputs:
     - src: camera source string (RTSP URL or numeric webcam index string)
     - name: camera label used for logs
     - capture_cfg: CaptureConfig (buffer_size, reconnect delays, max_read_failures)
   - State:
     - frame: last successfully decoded frame (OpenCV BGR ndarray)
     - lock: thread lock for safe read/write of frame
     - running/thread: lifecycle control

2. start()
   - Sets running=True
   - Starts a daemon thread executing _loop()
   - Returns self so caller can chain .start()

3. _open()
   - Converts src to int if src is digit-only string (e.g., "0")
   - Opens cv2.VideoCapture(src)
   - Applies cap.set(CAP_PROP_BUFFERSIZE, capture_cfg.buffer_size)
   - Returns opened handle (may still fail isOpened())

4. _loop() reconnect/read logic
   - Uses exponential-style backoff:
     - start: reconnect_initial_delay_sec
     - cap: reconnect_max_delay_sec
   - While running:
     a) open capture
     b) if open fails:
        - log cannot open
        - sleep(delay)
        - delay=min(delay*1.5, max_delay)
        - retry
     c) if open succeeds:
        - reset delay to initial
        - set failures=0
        - read frames in tight loop

5. Per-read behavior
   - cap.read() -> (ok, frame)
   - if not ok:
     - failures += 1
     - if failures > max_read_failures:
       - mark stream lost and break inner loop to reconnect
     - else sleep(0.01) and retry read
   - if ok:
     - failures=0
     - write latest frame under lock: self.frame = frame

6. Consumer side
   - get_frame() acquires lock and returns copy() of last frame
   - If never received frame yet, returns None
   - Returning copy prevents downstream mutation from corrupting shared latest frame

7. stop()
   - running=False
   - joins thread (timeout 3s)
   - _loop releases cap and exits


B) Pipeline Handoff: Frame to Person Track Candidates
-----------------------------------------------------
Implementation: production/src/crosscamreid/pipeline.py

1. Camera startup
   - run_app(...) creates RTSPCapture per configured camera and starts each capture thread
   - Main loop continuously polls latest frame for each camera

2. Per-frame detector/tracker call
   - For each available frame:
     result = pose.track(
       frame,
       persist=True,
       classes=[0],
       conf=config.gating.person_conf_thresh,
       tracker=config.runtime.tracker,
       verbose=False
     )[0]

3. Important meaning of detector arguments
   - persist=True: keeps tracker state across frames
   - classes=[0]: only person class
   - conf=person_conf_thresh: rejects low-confidence person detections before downstream processing
   - tracker=...yaml: tracker algorithm configuration (e.g., ByteTrack)

4. Required detector outputs for downstream processing
   Pipeline continues only if all are present:
   - result.boxes exists and non-empty
   - result.keypoints exists
   - result.boxes.id exists (track IDs)

5. Extracted per-person items
   - bbox: boxes.xyxy
   - tid: boxes.id (integer track id)
   - kp_xy: keypoint coordinates
   - kp_conf: keypoint confidences

6. Processor selection by camera role
   - MASTER -> process_master(...)
   - SLAVE  -> process_slave(...)


C) Gating Before Embedding
--------------------------
Implementation: production/src/crosscamreid/processor.py and keypoints.py

1. First gate: keypoint validity
   keypoint_gate requires confidence >= keypoint_conf_thresh for:
   - left_eye, right_eye, left_knee, right_knee

2. If keypoint gate fails
   - No embedding is computed
   - Record returned as UNKNOWN with keypoint_valid=False

3. Torso region computation
   torso_region_bbox uses shoulders + knees for region and applies padding:
   - region_pad_frac controls proportional expansion
   - min_region_side rejects too-small regions

4. Crop source decision for embedding
   - If backend.use_full_body == True -> use detector bbox
   - Else -> use torso region

Notes:
- fastreid backend sets use_full_body=True.
- onnx/tensorrt inherit default use_full_body=False, so torso region is used there.


D) Embedding Creation: Backend-Independent Steps
------------------------------------------------
Shared base implementation: production/src/crosscamreid/reid/base.py

1. Geometry clamp
   - bbox coords are clamped within frame bounds
   - ensures non-empty rectangle with at least 1 pixel width/height

2. Crop validity checks
   - Reject if crop is empty
   - Reject if crop height < 10 or width < 5
   - In such cases embed(...) returns None

3. Resize and normalization (_preprocess path used by ONNX/TensorRT)
   - Resize to input_hw (default 224x224)
   - BGR -> RGB
   - Convert to float32 in [0,1]
   - Normalize with ImageNet mean/std
   - Reorder HWC -> NCHW
   - Add batch dimension -> shape (1,3,H,W)

4. Postprocess (_postprocess for all backends)
   - Flatten feature tensor
   - Convert float32
   - L2 normalize
   - If norm < 1e-6, return None
   - Else return unit vector embedding


E) ONNX Runtime Embedding Path
------------------------------
Implementation: production/src/crosscamreid/reid/onnx_backend.py

1. Initialization
   - Creates InferenceSession(model_path, providers=[CUDAExecutionProvider, CPUExecutionProvider])
   - Reads input/output tensor names
   - Runs dummy input once to infer embedding dimension

2. embed(frame, bbox)
   - Calls _preprocess(...)
   - Session run: session.run([output_name], {input_name: inp})
   - Calls _postprocess(feat)
   - Returns normalized embedding or None


F) TensorRT Embedding Path
--------------------------
Implementation: production/src/crosscamreid/reid/tensorrt_backend.py

1. Initialization
   - Imports tensorrt + pycuda
   - Deserializes engine from file
   - Creates execution context and CUDA stream
   - Resolves one input + one output tensor
   - Allocates host/device buffers
   - Sets tensor addresses for async execution
   - Computes embedding dimension from output shape

2. embed(frame, bbox)
   - _preprocess(...)
   - copy host input
   - memcpy host->device async
   - execute_async_v3 on context
   - memcpy device->host async
   - stream synchronize
   - _postprocess(host_output)


G) FastReID Embedding Path
--------------------------
Implementation: production/src/crosscamreid/reid/fastreid_backend.py

1. Initialization
   - Adds fast-reid repo root to sys.path
   - Loads cfg from fastreid config yaml
   - Assigns MODEL.WEIGHTS
   - Resolves device (falls back to CPU if CUDA unavailable)
   - Builds fastreid DefaultPredictor(cfg)
   - Reads INPUT.SIZE_TEST from cfg
   - Runs dummy tensor once to infer embedding dimension

2. embed(frame, bbox)
   - Clamps bbox and validates crop size (same style checks)
   - BGR -> RGB
   - Resize to cfg INPUT.SIZE_TEST
   - Convert to float32 tensor NCHW + batch
   - predictor(tensor) produces feature tensor
   - _postprocess(feat) applies L2 normalization

3. Why this path differs from ONNX/TensorRT
   - fastreid backend does NOT apply base _preprocess mean/std externally
   - FastReID model performs pixel mean/std normalization internally in Baseline.preprocess_image()
   - Therefore backend feeds raw float32 RGB [0..255] tensor to predictor


H) Exact Conditions That Prevent Embedding Generation
-----------------------------------------------------
Embedding is skipped (or returns None) in any of these cases:
1. No frame available from capture yet
2. Detector produced no valid person boxes/ids/keypoints for that frame
3. Keypoint gate failed
4. Selected crop bbox is None (e.g., torso region invalid for non-full-body backend)
5. Crop empty or too small (<10 px height or <5 px width)
6. Backend inference failure (e.g., TensorRT execute returns false)
7. Feature norm too small (<1e-6) after inference


I) What This Report Excludes
----------------------------
This report intentionally stops at embedding generation. It does not detail:
- SID matching thresholds
- enrollment/qualification counters
- database write policy
Those are downstream identity-assignment stages after embeddings are already produced.


J) Practical Interpretation
---------------------------
If you need consistent embeddings, the highest-impact controls before model inference are:
- capture stability (read failures/reconnect)
- person confidence threshold
- keypoint confidence threshold
- crop policy (full body vs torso)
- minimum crop size constraints
- backend-specific preprocessing expectations

All of these are enforced before the final embedding vector is emitted.
